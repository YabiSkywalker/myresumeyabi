Achieving High Availability

High availability is characterized by a system's ability to operate with little to no downtime. A highly available platform may not necessarily look like the architecture above. As always the business need drives the solution; therefore, even a simple autoscaling group with a load balancer in front of it could be considered highly available - so long as it successfully delivers the underlying content through unexpected events.

High Availability vs Fault Tolerance

All highly available systems incorporate some degree of redundancy. Redundancy is what makes a system Fault Tolerant. Typically, you'll see redundancy implemented in key components such as servers, and databases.

For a system to be highly available it must be fully operational at all times with little to no failures. On the other hand, when a system is fault-tolerant it means that it will withstand failures.

As the system grows, we often split it into tiers (also known as layers), which sets it on the path of greater scale and availability.

This use case:

In this instance we have a microservice running within an Auto Scaling Group (ASG) fronted by a load balancer (LB). The microservice does not need to access the internet as it's primary function is to log entries into a database. While it is not necessary, there are NAT gateways configured within public subnets to provide the application internet access. Internet access is provided by an Internet Gateway (IGW), which the NAT gateways use. The ASG is configured to expand, and contract based off limits specified within the terraform modules. In addition to this there is an Aurora database configured with asynchronous multi-az replication to provide redundancy.
